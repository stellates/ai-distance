# CODING_AGENT_SPEC.md

## 目的

このエージェントは、
「AIとのちょうどいい距離感」の新記事を、
既存シリーズとの整合性を保ちながら、
ローカル作業ツリー上で自動生成し、commit 可能な状態まで整える。

入力は以下のみ：

- テーマ
- 任意の補足メモ

エージェントは以下を自動で行う：

- 過去記事調査
- 記事番号採番
- タイトル生成
- 漫画構成生成
- 画像生成
- 画像品質レビュー
- HTML生成
- index.html 更新
- git commit

ただし、`git push`、PR作成、`main` への merge、GitHub Pages の本番反映確認は人間が行う。

---

# 基本方針

- 人間への確認待ちは原則禁止
- 必要な判断は既存記事から推定すること
- シリーズの雰囲気維持を最優先すること
- 過剰に説明的にしない
- 説教臭くしない
- 「AIを怖がる」のではなく「ちょうどいい距離感」を重視する
- 作業完了後は commit までで停止し、push や公開操作は行わない

---

# 作業開始時の必須確認

作業開始時に以下を確認すること。

- 最新3記事
- index.html
- assets/
- posts/

styles.css は参照のみ可能。
変更は禁止。

確認対象：

- タイトル長
- 情報密度
- 漫画テンポ
- キャラクターデザイン
- 色味
- 教育感
- UIレイアウト
- モバイル表示前提の余白感

---

# 記事番号ルール

- posts/*.html を確認
- 最大番号 + 1 を新記事番号とする
- 3桁ゼロ埋め

例：

- 001.html
- 014.html
- 125.html

---

# 記事タイトルルール

タイトルは以下を満たすこと。

- 短め
- 一目でテーマ理解可能
- 不安を煽りすぎない
- SNSで流し見して意味が伝わる
- 「あるある感」を優先

避けること：

- 極端な煽り
- 陰謀論的表現
- 専門用語過多
- 長文タイトル

---

# 漫画構成ルール

画像は1枚完結型の縦長漫画とする。

以下を満たすこと：

- スマホ閲覧前提
- 直感的に読める
- コマ順が自然
- セリフ量を詰め込みすぎない
- 1コマ1メッセージを基本とする
- 最後に軽い学びを置く
- 説教で終わらない

---

# シリーズ整合性ルール

既存記事と以下を合わせること：

- 色味
- 温度感
- キャラの親しみやすさ
- 情報密度
- 余白感
- 教育感
- タイトル配置
- フォントサイズ感

シリーズから極端に浮くデザインは禁止。

---

# 画像生成仕様

画像は以下を満たすこと。

- 日本語漫画形式
- 縦長
- SNS共有向け
- 日本語可読性を重視
- テキスト量を増やしすぎない
- 親しみやすいキャラクター
- 軽い自虐感
- 教育感あり
- 過去記事と近いトーン

タイトル文字は視認可能にすること。

## 画像生成モデル

- ChatGPT / Codex 環境で利用可能な画像生成機能を利用すること
- 外部画像生成サービスの利用は禁止
- シリーズ既存画像と大きく異なる画風は禁止

---

# 画像レビュー

画像生成後、必ずレビューを行う。

生成した画像ファイルを必ず再読込し、
Vision機能で実画像レビューを行うこと。

確認項目：

- 日本語が読める
- 文字化けしていない
- 意味不明な英字がない
- 吹き出しが切れていない
- コマ順が自然
- テーマが反映されている
- 情報量が過密でない
- キャラ崩壊していない
- 指や顔が極端に崩壊していない
- タイトルが視認可能
- 過去記事と比較して違和感が強すぎない

問題があれば再生成すること。

最大3回まで再生成可能。

---

# 再生成条件

## レビュー優先順位

以下を優先して品質確認すること：

1. 日本語可読性
2. テーマ一致
3. コマ順の自然さ
4. シリーズ整合性
5. ビジュアル品質

以下の場合は失敗とみなし再生成すること：

- 日本語が読めない
- 意味不明な文字列
- ノイズ文字
- コマ順破綻
- 情報量過多
- タイトル欠落
- キャラ崩壊
- 構図崩壊
- テーマ反映不足

---

# 画像ファイル名ルール

以下形式：

NNN-kebab-case-topic.png

例：

- 014-fake-paper-citation.png
- 015-overconfident-ai.png

英単語は3〜5語程度に抑えること。

---

# HTML生成ルール

新記事HTMLを posts/NNN.html に生成する。

必須条件：

- 既存HTML構造を踏襲
- class名変更禁止
- モバイル表示維持
- OGP構造維持
- メタタグ構造維持
- 相対パス維持
- 前後記事リンク更新
- 新記事ページの og:image を新記事画像へ更新すること

最新記事を複製して作業開始すること。
既存記事をテンプレートとして利用すること。
画像は既存記事と同じディレクトリへ保存すること。

---

# index.html 更新ルール

- 新記事カードは先頭に追加
- 一覧は新しい順
- post-number を一致させる
- 既存構造変更禁止
- カードデザイン変更禁止
- 既存記事カード変更禁止
- 既存記事順序変更禁止

---

# 禁止事項

以下は禁止：

- styles.css の変更
- README.md の変更
- LICENSE の変更
- 既存記事本文改変
- HTML構造変更
- CSS設計変更
- class名変更
- ディレクトリ構成変更
- git add 対象外ファイルを作らない
- git push の実行
- PR作成、PR merge、auto-merge の実行
- GitHub Pages 本番環境への反映操作

---

# Git運用

作業は、現在のローカル作業ブランチ上で行う。

通常は `main` で直接作業してよいが、作業完了後は commit までで停止すること。
`git push` は人間が、ローカル確認後に実行する。

作業完了後、以下を実行すること。

```bash
git status
git add .
git commit -m "add post NNN: short-topic"
```

commit 後、以下を出力して停止すること。

- commit hash
- 変更ファイル一覧
- 人間向けローカル確認手順
- push は未実行であること

---

# 停止条件

以下の場合は作業停止すること：

- 画像生成3回失敗
- HTML構造維持不能
- index.html 更新失敗
- パス解決不能
- git conflict 発生
- push、PR作成、merge、公開操作が必要になった場合

停止時は原因を出力すること。

---

# 完成条件

以下をすべて満たした場合のみ完了とする。

- 新記事HTML存在
- index.html更新済み
- 前後記事リンク正常
- 画像生成済み
- 画像レビュー通過
- モバイル表示崩れなし
- GitHub Pages 公開可能状態
- git commit成功
- git push 未実行
- 人間向けローカル確認手順を出力済み

---

# エージェントへの追加指示

- 必要以上に人間へ質問しない
- 判断不能な場合は既存記事に合わせる
- テーマは「一般ユーザーの日常あるある」に変換して描写する
- シリーズの一貫性を優先する
- 「ちょっと笑える教育漫画」を維持する
- AI否定ではなく「付き合い方」をテーマにする
- commit 後は、push せずに人間へ確認を促す
