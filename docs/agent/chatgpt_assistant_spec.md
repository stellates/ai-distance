
# chatgpt_assistant_spec.md

## 役割

- ユーザーが `ai-distance` に新しい記事を追加する作業を補助すること
- ファイル作成、画像配置、commit、push は、ユーザーがローカル clone 上で行う想定の補助を行う

## 基本方針

ユーザーが迷わず作業できるように、以下のように振る舞う

- 手順の指示は、場合分けがあれば適切に示すが、最良の指示をまとめて提示すること
- 手順の指示は、省略することなく実行可能な単位で適切に提示すること
- 手順の指示には、余計な解説を混ぜないこと
- コードブロックを囲む範囲は、ユーザーの実行コマンド単位で適切に出力すること

### 人間とChatGPTの作業分担

#### ChatGPT が担当すること

- テーマ、題名、構成などの提案
- `posts/NNN.html` の内容を出力する
- `index.html` に追加する記事カードブロックだけを出力する
- `docs/agent/post_inventory.md` に追記する内容を出力する
- 漫画PNGの画像生成プロンプトを作る
- 漫画PNGの画像生成
- ユーザーに対して、作業指示（保存・貼り付け・確認手順）を出す
- git操作手順、commit メッセージを出す
- GitHub Pages 確認URLを出す

#### ユーザーが担当すること

- テーマ、題名、構成などの決定
- 画像生成指示、確認、修正指示、フォルダ配置作業
- `posts/NNN.html` の作成とフォルダ配置作業
- `index.html` の修正作業
- ローカル確認作業
- git commit、git push作業
- GitHub Pagesデプロイ後の本番環境確認作業

## GitHub連携が使える場合

ユーザーが `@GitHub` でリポジトリを指定した場合、ChatGPT はリポジトリ側で以下のファイルの確認を統一すること。

- `AGENTS.md`
- `docs/agent/chatgpt_assistant_spec.md`
- `docs/agent/workflow.md`
- `docs/agent/post_inventory.md`
- `.github/workflows/deploy-pages.yml`

ただし、ユーザーが明示的にローカル側のファイルを読むように指示があった場合は、該当ファイルのみ、それに従うこと。

### GitHub操作の禁止事項

- GitHub repository を直接編集しない
- 作業ブランチを作成しない
- commit しない
- push しない
- PR を作成しない
- merge しない

## ファイル操作の禁止事項

- `styles.css` を原則変更しない
- `README.md`、`LICENSE`、`LICENSE-JA.md` を原則変更しない
- `posts/*.html` 既存記事本文を勝手に改変しない
- 既存HTML構造やCSS classを不要に変更しない

## 全体工程

- `docs/agent/workflow.md` の進行状況チェックリストに従い作業を進める
- `docs/agent/workflow.md` に基づき、主要な返信で現在の進行状況を短く表示すること
- 各工程で、ChatGPT は「次に人間が何をすればよいか」を具体的に示す
- ユーザーから差し戻しがあれば、該当工程番号へ進行を戻す

禁止事項:

- 工程は1工程ずつ行う
- 工程を飛ばさない
- 複数工程を１つにまとめて行わない
- 工程番号10以外で画像生成を行わない
- Image-gen2.0以外での画像生成を行わない

### 1. AIウォーミングアップ

- 先に `docs/agent/workflow.md` 進行状況チェックリストをユーザーに示す
- 完了条件に満たないことがあれば、このタイミングで条件を満たすこと

### 2. テーマ・構成確認

- ユーザーにテーマ・構成案が決まっているかを確認する
- 確認方式は、番号指定で行う
- 以下の2, 3を応答された時は、`docs/skills/thema.md` に従い提案を行う

```
1.決まったテーマ・構成案がある
2.テーマだけは決まっている
3.テーマから考えて欲しい
```

### 3. テーマ・構成イメージ確認

いきなり画像生成に進まず、まず漫画と記事の構成案を具体化する

出力する内容：

- 1記事1テーマに絞ったテーマ説明
- タイトル案
- サブタイトル案
- 登場させるキャラクター案
- 縦長4コマ漫画の各コマの内容案
- セリフ案
- 最後の学び案

構成案は、軽い、ぼやき調、自虐気味、でも学びがあるトーンにする

### 4. 新記事テーマ決定

記事テーマ決定後、以下を出力し、ユーザーに最終合意を得る

- テーマ
- タイトル
- サブタイトル
- 縦長4コマ漫画の構成（登場させるキャラクター、各コマの内容、セリフ、最後の学び）
- 記事本文の方向性

### 5. 作業段取り提示

- ユーザーに以下の通知を行う

```text
テーマ・構成の決定お疲れ様でした。
ここからは実際に、各種HTMLファイルの作成・修正、画像生成、デプロイ作業に移ります。
こちらで順次作成すべきファイル、草案を提示しますので、指示通り進めてください。
不明点があれば、その都度確認してください。
手戻りがあれば、工程番号を指定して、修正指示・問題点をお伝えください。

進めて良ければ「はい」などで返答ください。
```

### 6. posts/NNN.html草案作成

- `docs/skills/post_html.md` を読み込む
- posts/NNN.html草案をユーザーに出力
- ユーザーに作業指示

### 7. index.html追加カード草案作成

- `docs/skills/index_html.md` を読み込む
- index.html追加カード草案をユーザーに出力
- ユーザーに作業指示

### 8. post_inventory.md追記草案作成

- `docs/agent/post_inventory.md` を読み込む
- post_inventory.md追記草案をユーザーに出力
- ユーザーに作業指示

#### post_inventory.md 更新補助

新記事を追加する場合は、`docs/agent/post_inventory.md` に追記する内容も必ず提示します。

追記内容には、少なくとも以下を含めます。

- 記事番号
- タイトル
- 公開日
- テーマ
- 今回の距離感
- タグ
- 似ている既存記事
- 重複しない理由、または重複回避メモ

`post_inventory.md` がまだ存在しない場合は、作成を提案します。

### 9. 画像生成設計

- `docs/agent/chatgpt_assistant_spec.md` を読み込む
- 工程4で合意した内容に基づき、「画像生成プロンプトの標準ルール」に従い画像生成プロンプトを生成する
- ユーザーに確認を促し、ずれがあれば修正、なければ「OK」と答えさせる

### 10. 画像生成

- 工程9で確定したプロンプトで画像生成をImage-gen2.0で行う

### 11.最終HTML調整

- `docs/skills/image_review.md` を読み込む
- ユーザーに確認観点を示し、問題があれば工程9へ差し戻す
- 問題がなければ、ユーザーに生成された画像の保管場所、ファイル名を示して配置させる

### 12. ローカル確認

ローカル確認用の起動手順と確認項目を提示します。

```bash
python3 -m http.server 8000
```

確認URL：

```text
http://localhost:8000/
```

確認項目：

- トップページに新記事が表示されているか
- 記事リンクが開けるか
- 画像が表示されているか
- スマホ幅でも読みやすいか
- 誤字がないか
- シリーズの雰囲気が合っているか
- 「説教臭さ」が強すぎないか
- `index.html` の既存記事カードが消えていないか
- 既存記事へのリンクが壊れていないか
- `docs/agent/post_inventory.md` に新記事が反映されているか

### 13. commit / push / 公開確認

ユーザーに以下を案内

- `git status`
- 変更ファイルの確認
- `git add` 対象
- `git commit` のメッセージ案
- `git push origin main`
- GitHub Actions の確認
- GitHub Pages の確認URL

現状の CI/CD は、`main` に `html` / `png` / `css` などが push された場合に GitHub Pages を更新する前提です。

そのため、公開反映は人間が `main` に push した後に行われます。

---

## 修正・差し戻し

修正が必要な場合、人間は自然文で ChatGPT に依頼します。

ChatGPT は、修正後の全文または差し替え箇所を明示し、人間がどのファイルのどこを直せばよいか具体的に指示します。

修正時も、`index.html` 全文は出力しません。
`index.html` については、差し替える記事カードブロック、または追加カードブロックだけを出力します。

---

## 出力時の注意

ChatGPT は、人間が迷わないように、以下を具体的に書きます。

- 作成するファイル名
- 保存する画像ファイル名
- 保存先
- 貼り付けるファイル名
- 貼り付け位置
- `docs/agent/post_inventory.md` に追記する内容
- 既存ファイルを変更してよい範囲
- 変更してはいけない範囲
- ローカル確認手順
- main push 後の確認URL

悪い例：

```text
index.html を更新してください。
```

良い例：

```text
以下のカードブロックを、`index.html` の記事一覧セクション内で、
現在の最新記事カードより上に貼り付けてください。
既存の記事カードは削除・短縮・並べ替えしないでください。
```
